# Product Guidelines — @pleaseai/spring-docs

> Style and content principles for code, generated Markdown, and project communication.

## Content Conversion Principles

Generated Markdown is the product. These rules govern AsciiDoc → Markdown conversion (`scripts/lib/antora-rules.ts`).

### 1. Preserve Meaning, Change Format Only

- The semantic content of upstream documentation must remain unchanged.
- Format conversions are mechanical: `xref:`, `include::`, admonitions, tab blocks, attribute substitution.
- **Never** paraphrase, summarize, or "improve" upstream prose.
- If meaning cannot be preserved (e.g., AsciiDoc feature with no Markdown equivalent), prefer an explicit fenced block with the original syntax over a lossy approximation.

### 2. LLM-Friendly Output

The target consumer is an LLM, not a human browser. Optimize for token efficiency and unambiguous parsing.

| Do | Don't |
|---|---|
| Use GFM admonitions (`> [!NOTE]`) | Use raw HTML `<div class="admonitionblock">` |
| Inline `include::` content at conversion time | Keep `include::` directives that require a runtime resolver |
| Resolve AsciiDoc attribute substitution (`{spring-version}` → literal value) | Pass attribute placeholders through |
| Tab-block code samples → headed fences (`### Gradle` + ` ```kotlin`) | Tab UI widget HTML |
| Relative Markdown links (`./web/mvc.md`) | Absolute URLs to upstream HTML |

### 3. Deterministic Conversion

- Identical upstream commit → identical Markdown output, byte for byte.
- No timestamps, run IDs, or environment metadata in converted content. (Those live in `manifest.json` only.)
- Sort attributes, multi-value fields, and link lists in a stable order.
- Line endings: LF only.

### 4. Validation Gates

Every conversion output must pass before publishing:

- Markdown link check (internal links resolve; external links return 2xx/3xx in audit mode)
- Frontmatter schema check (where frontmatter applies)
- Size sanity check (output within ±25% of historical baseline per project)

## Repository Code Style

### TypeScript / Bun

- ESM only. No CommonJS interop in new code.
- `bun run` for scripts; `tsc --noEmit` for type-checking only (Bun executes TS directly).
- Strict TypeScript (`strict: true`, `noUncheckedIndexedAccess: true`).
- Prefer `URL` over string concatenation for paths and URIs.
- Use `node:fs/promises` (or Bun's built-ins) — no callback APIs.

### Naming

- Scripts: kebab-case verb-first (`fetch-upstream.ts`, `package-release.ts`).
- Library modules: kebab-case noun (`antora-rules.ts`, `manifest.ts`).
- Types: PascalCase. Interfaces named for the thing, not prefixed with `I`.
- Constants: `UPPER_SNAKE_CASE` only for true constants; otherwise camelCase.

### Error Handling

- Conversion errors must include the upstream file path and line number when known.
- Pipeline failures exit with non-zero status and a one-line summary, with detailed diagnostics on stderr.
- No silent fallbacks. If a conversion rule encounters unhandled AsciiDoc syntax, fail loudly and add a rule, do not paper over it.

## Documentation Style (this repo)

Writing for `README.md`, ADRs, track specs, and other internal docs.

### Voice & Tone

- **Direct, factual, no marketing.** Describe what the system does, not how exciting it is.
- Second person ("you") for instructions, third person for descriptions.
- Present tense for current behavior, future tense only for explicitly forthcoming work.
- Active voice. Avoid "is generated by", prefer "the pipeline generates".

### Structure

- Lead with the takeaway. Tables and code blocks before long prose.
- Use **What this is** / **What this is NOT** framing when the project is easy to misclassify (already in README).
- ADRs follow `0NNN-kebab-title.md` naming under `.please/docs/decisions/`.
- Track specs follow the standard `spec.md` / `plan.md` template inside `.please/docs/tracks/active/{slug}-{date}/`.

### Citations & Attribution

- Always link upstream sources when discussing conversion fixes or content provenance.
- License notices in archives must reference both Apache-2.0 and the specific upstream repo.
- Never silently re-host third-party content without a `NOTICE` entry.

## Commit & PR Conventions

See [`commit-convention.md`](./commit-convention.md) when initialized via `/standards:init-rules`. Defaults:

- Conventional Commits format (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `perf:`, `test:`, `ci:`, `build:`).
- Header ≤ 100 chars, lowercase type, imperative mood.
- Scope when useful: `feat(convert): ...`, `fix(antora-rules): ...`, `ci(matrix-build): ...`.
- Reference issues in the footer: `Refs: #123` or `Closes: #123`.

## CLI & Script UX

The conversion pipeline is operator-facing. Scripts should:

- Print a one-line summary on success ("Converted 142 files to /tmp/dist/spring-framework-6.2.0/").
- Stream progress for long operations (per-project, per-version).
- Accept `--out`, `--project`, `--version` as standard flags.
- Provide `--dry-run` for any operation that writes outside `dist/` or `node_modules/`.
- Fail with actionable errors (path to offending file, suggested fix or upstream link).

## Release & Public Communication

- GitHub Release notes describe what's in the archive, not changes to the converter (those go in PR descriptions / changelog).
- Tag names are immutable contracts: `framework-6.2.0` means "the docs for spring-framework v6.2.0", forever.
- Rebuilds use suffix tags (`framework-6.2.0+rebuild.1`); `catalog.json` points consumers at the latest.
- When responding to upstream Spring maintainer concerns, lead with "we want to remain a good downstream citizen" — accommodate, don't argue.
